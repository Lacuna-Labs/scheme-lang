1. What's Sakura Scheme?

Sakura Scheme is a small Lisp dialect that runs in the browser. You use it to write carts — short, state-machine-shaped automations that do useful work against your shop: audit listings, draft replies, paint a similarity grid, queue an envelope for the morning. One cart, one job.

The reason it's a Lisp and not a config file is that real automations have branches, retries, and rollback. A list of YAML steps can't say "if the model returned rate-limited, sleep thirty seconds and try again, and if quota is exhausted, surface a clear error to the operator." Sakura Scheme can say that in three lines, and the cart driver knows what to do with each descriptor. Here's the shortest cart that actually works:

;;~ title    "Hello, Sakura"
;;~ author   "you"
;;~ version  1
;;~ mode     analysis
;;~ flavor   white
;;~ id       hello-sakura
;;~ touches  ()
;;~ summary  "Smallest cart that runs end-to-end."

(cart 'hello-sakura
  '((author    . "you")
    (version   . 1)
    (read-only . #t)))

(define (start ctx)
  (card-emit 'engine 'hello "🌸 hello from Sakura Scheme")
  (done))

That's it. Eight lines of header, three lines of code, one cart. The rest of this tutorial is everything you need to grow from that into the carts that ship in production today.

Try it. The little editor below runs real Sakura Scheme in your browser — no install, no sign-in. Press Run to evaluate the expression. You can change it; the engine will tell you what it returns.

2. Setup — where you write

Carts live on disk. The directory tree is the corpus:

curator-web/src/scheme/carts/
  pink/      ;; free tier — Sakura on-device (1.7B router), no cloud spend
  imagine/   ;; Imagine $9.99 — 8B on Fly (Engram)
  dream/     ;; Dream $39.99 — cloud reasoning tier; maps to light-purple
  magic/     ;; Magic $99.99 — deep reasoning tier; maps to deep-purple
  etsy/      ;; hand-curated marketplace verbs (do not auto-regen)
  google/    ;; hand-curated platform verbs
  scenes/    ;; named visual scenes
  transfer/  ;; cross-shop transfers

Pick the tier that matches what the cart costs to run, then create a file named after the cart's slug: your-slug.sks. The slug is the filename, the id in the header, and the symbol in the (cart 'slug …) form — they all match.

Lint gate — Star 1

Before any cart runs, cartLint.js reads the source and checks the Star 1 invariants: the (cart 'id …) header exists, a (define (start ctx) …) entry state is present, every (act 'verb args 'on-result) uses a quoted symbol as the resume target (never an inline lambda), every (next 'X ctx) and (after N 'X ctx) target is actually defined, and every (escalate 'kind …) uses a quoted symbol. Carts that fail Star 1 never reach the driver.

When you add, rename, or delete a .sks file, three indexes need to be regenerated so the rest of the system sees your cart:

npm --prefix curator-web run build:cart-index

That single command rebuilds index.json (the searchable index), sakura-corpus.jsonl (the intent-to-slug pairs the on-device model learns from), and sakura-cart-breadcrumbs.json (the natural-language lookup). It's deterministic — running it twice on unchanged disk is a no-op.

3. The four-block cart shape

Every cart has the same four blocks, in the same order. Memorise the shape once and you'll never have to think about the skeleton again.

  1. Sled — the ;;~ key value metadata header. Tooling parses it.
  2. Doc block — what / why / idempotency / error grammar, in operator words.
  3. Cart registration — (cart 'slug '((…))).
  4. State functions — (define (start ctx) …) and friends.

Here are the first eighteen lines of a real shipping cart, after-hours-acknowledge, annotated:

;;~ title    "After-hours message acknowledgement"   ; (1) sled — tooling reads this
;;~ author   "lacuna-engineering"
;;~ version  1
;;~ mode     automation                              ; analysis | automation
;;~ flavor   pink                                    ; the dot tier indicator
;;~ id       after-hours-acknowledge                 ; matches filename + (cart 'slug …)
;;~ trigger  event:conversation-received             ; cron:daily | event:* | on-demand
;;~ touches  (cortex)                                ; () = read-only; what it writes
;;~ summary  "Reply instantly after hours."

;; ── after-hours-acknowledge · event-triggered cart ─────────────  (2) doc block
;; WHAT:  When a buyer messages outside hours, send a warm ack so no
;;        buyer ever feels ghosted.
;; WHY:   After-hours silence is the top driver of 1-star reviews.
;; ERROR GRAMMAR:
;;   'state-blocks-spend  loam said vacation / paused.
;;   'within-shop-hours   event fired inside open window; skip cleanly.
;;   'send-failed         marketplace reply write returned non-ok.

(cart 'after-hours-acknowledge                       ; (3) registration
  '((author    . "lacuna-engineering")
    (version   . 1)
    (cadence   . event:conversation-received)
    (tier      . free)
    (model     . model/fast)
    (read-only . #f)))                               ; #f = this cart writes

;; ── state functions ──────────────────────────────────────────    (4) the state machine

(define (start ctx)
  (act 'loam/operator-state (list) 'check-state))

The sled is read by the studio UI, the linter, the corpus pipeline, and the search index. If you omit a field, the cart silently drops off those surfaces — you'll think it's broken and it'll just be invisible. Always write all nine sled lines.

4. Atoms, verbs, and capability-first names

The vocabulary inside a cart falls into three groups.

Primitives are the verbs the engine implements natively in JavaScript. There are about fifteen, in four namespaces: motion/, note/, surface/, card/. You call them, you don't define them. Example: (card-emit 'engine 'hello "🌸").

Macros are operator-facing compositions that expand to primitives. motion/glide expands to motion/move-to with a named curve and a duration. form/12-bar-blues expands to a sequence of note/place-at calls. Use macros; they read better and they keep the cart honest about what it actually does.

Capability verbs are how a cart asks the runtime to do something on the operator's behalf — fetch listings, score embeddings, call a model. They use the namespace/capability shape and they never name a vendor. Five examples you'll see in almost every cart:

(cortex/recall      '(:topic shop-hours :window now))     ; on-device memory read
(cortex/cosine-topk 'anchor-id 8 '(:topic embeddings))    ; on-device vector match
(model/fast         "prompt" :budget 'fast)               ; low-latency model call
(etsy/listings      '(active))                            ; marketplace read
(etsy/conversations '(:reply convo-id "Hi there"))        ; marketplace write

Try an atom. Atoms are the simplest scheme values — symbols, numbers, lists. Push Run; change the value; learn the printed shape.

These same atoms and lists — symbols, keywords, numbers, strings — are the pieces that make up a slat. When you later see a labeled list (event :ts … :kind …) flying between two carts, it's this same vocabulary. See §L Slats.

The reason the verbs are capability-first — (model/fast …) instead of the vendor name — is that the backend swaps. The operator never sees vendor names; the cart never hard-codes them. When the cascade picks a new workhorse next quarter, every cart that wrote (model/fast …) keeps working unchanged.

Pending — be honest

Some capability verbs aren't wired on every device. If (act 'etsy/conversations …) returns null or 'not-yet-wired, the cart must (escalate 'service-not-yet-wired …) — never silently succeed on a no-op. The runtime catches the escalate and surfaces a real error; fake success is a sue-able product claim.

5. State machines — start, next, done, escalate

A cart is a state machine. Each (define (state-name ctx) …) is one node. The function returns a driver descriptor that tells the cart driver what to do next. There are seven descriptors:

(next 'state ctx)              ; advance to another state, carry ctx forward
(done)                         ; finished, no more work
(escalate 'kind detail)        ; pause and surface; the host decides
(after N 'state ctx)           ; sleep N seconds, resume at 'state
(act 'verb args 'on-result)    ; one tool call, resume at 'on-result
(wait 'event)                  ; block until a named event fires
(interrupted 'reason ctx)      ; live-voice mid-speech catch

Try a state-machine move. The descriptors are ordinary Scheme values — you can build one and print it. Press Run.

You navigate ctx with three helpers: (ctx-get 'key ctx) reads a value the surface put in, (ctx-set 'key value ctx) returns a new ctx with that key set, and (ctx-result ctx) reads the response from the last (act …) call.

Whatever you put on ctx is a value. When the value has to leave the cart — go to Cortex, land in a mailbox, get logged for later — it travels as a slat. See §L Slats.

Here's a three-state cart that pulls one piece of data, validates it, and announces:

;; entry — start with one act call
(define (start ctx)
  (act 'cortex/recall
       (list '(:topic todays-orders :window now))
       'check-orders))

;; resume — the act's response landed in (ctx-result ctx)
(define (check-orders ctx)
  (let ((orders (ctx-result ctx)))
    (cond
      ((null? orders)
        (escalate 'cortex-not-ready
                  '(:status not-wired :reason "no-orders-topic-in-cortex")))
      ((= (length orders) 0)
        (next 'announce-empty ctx))
      (else
        (next 'announce-count (ctx-set 'orders orders ctx))))))

;; terminal — surface the result, then done
(define (announce-count ctx)
  (let ((n (length (ctx-get 'orders ctx))))
    (card-emit 'engine 'todays-order-count n)
    (done)))

(define (announce-empty ctx)
  (card-emit 'engine 'todays-order-count 0)
  (done))

Three rules to internalise:

  1. The 'on-result slot of (act …) is a quoted symbol, never an inline lambda. The cart is serialised, replayed, and audited; lambdas break all three.
  2. Every (escalate 'kind …) uses a quoted symbol for the kind. The host has a closed switch over these symbols and translates them into operator-facing copy.
  3. Every named state is either reachable from start or removed. Dead states are a Star 1 warning.

6. Cortex — recall, cosine-topk, remember

Cortex is the on-device memory layer. Three capability verbs cover almost everything a cart needs.

(cortex/recall key) reads a fact keyed by a topic. The key is a property list — :topic identifies what you want, :window scopes it in time. Returns null if nothing matches. Carts treat null as "not configured yet" and either fall back to a default or escalate honestly.

(cortex/cosine-topk anchor k topic) runs vector similarity against an embedding index, entirely on-device. Used for "find listings like this one", "find the closest past reply", "find the nearest tag." No network hop, no LLM spend.

(cortex/remember key value) writes a fact. Idempotency keys typically include a timestamp window so the same event re-firing today doesn't write twice.

The value Cortex reads back is a slat — the same shape you'd see if you opened Cortex's log directly. See §L Slats for how carts send data to each other in that shape, and the round-trip property that makes (cortex/recall) hand back the value (cortex/remember) put in.

From pink-listing-similarity, the on-device similarity grid:

;; pull the embedding-index header so we can gate on catalog size
(define (load-index ctx)
  (act 'cortex/recall
       (list '(:topic listing-embeddings-meta :window now))
       'check-index))

;; if there's enough catalog, run the cosine query
(define (fetch-topk ctx)
  (let* ((anchor (ctx-get 'anchor-id ctx))
         (k      (or (ctx-get 'similarity-topk ctx) 8)))
    (act 'cortex/cosine-topk
         (list anchor k '(:topic listing-embeddings))
         'check-topk)))

Notice how each Cortex call is wrapped in (act …) with a quoted resume symbol. That's the rule: every external read or write goes through (act …), so the runtime can audit it, retry it, and replay it.

7. Paint primitives — live

Carts paint to the canvas. The engine exposes a small set of paint verbs that take a named surface and a payload, and draw without you having to think about pixels. (paint-grid 'name rows columns) renders a table, (paint-grid-3state surface palette) walks a cellular grid and colours each live cell.

From the same similarity cart, here's the paint call that actually puts the matches on screen:

(define (render ctx)
  (let ((matches (ctx-get 'matches ctx))
        (anchor  (ctx-get 'anchor-id ctx)))
    (paint-grid 'listing-similarity-grid matches
                '(listing-id title score))
    (card-emit 'surface 'similar-to
               (list ':anchor anchor ':matches matches))
    (done)))

The two ideas the live demo is teaching you: addressable cells (every dot has a row, column, and an operator-meaningful value) and palette-driven colour (the cart never picks RGB; it picks a palette name and lets the surface decide). Real carts use the same shape, just with real data.

8. Sprite verbs

Sakura's sprite helpers — the sixteen colour-named dot-matrix workers, with Blossom (the pink one) front-and-centre — are addressed through the (sprite …) dispatcher. You don't poke their internals; you hand them an atom and the engine lowers it to a real card move, a flash, or a structured pending-visual envelope.

The atoms that ship today:

(sprite 'blossom (carry "listing-a" 50 75))     ; pick up + move a card
(sprite 'blossom (hand-to 'rose '("listing-a"))) ; pass to another sprite
(sprite 'mint    (set-down 'shelf-b "listing-a")) ; place at an address
(sprite 'rose    (flash "listing-a"))             ; glow on a target
(sprite 'mint    (size 2.0))                      ; scale (clamped 0.5–3.0)
(sprite 'mint    (turn 90))                       ; rotate (clamped 0–360)
(sprite 'rose    (in-order (turn 90) (flash "a")))

Each call returns one of two things. If the atom maps to a real engine verb (carry lowers to move-card, flash lowers to card-effect) the dispatch is real and the canvas reflects it. If the atom needs visual machinery that isn't there yet, the dispatcher emits its intent on the logbus and returns ['pending-visual', {…}] — an honest envelope, never a fake 'ok.

The live editor above runs the real sprite dispatcher. Try a different verb: (sprite 'mint 'flash) or (sprite 'rose 'wander). Wrong atoms surface their own error — the engine never lies about a missing wire.

The live preview is a stand-in: Blossom (pink) lifts a card, glides it from the left address to the right address, and sets it down. Your cart calls one line — (sprite 'blossom (carry "listing-a" 50 75)) — and the engine handles the timing curve, the lift, and the settle.

9. Your first full cart, end to end

Let's write one. The brief: an operator pastes a paragraph from a listing description and wants five scannable bullets. One model call, no Cortex write, no marketplace touch. Imagine-tier (the operator pays for the model call). This is a real shipping cart; bullet-points-from-paragraph.sks. We'll walk through writing it from a blank file.

Step 1 — pick the tier directory and slug

One fast model call, no on-device-only constraint, no big cascade — that's imagine tier. Create the file:

curator-web/src/scheme/carts/imagine/bullet-points-from-paragraph.sks

Step 2 — write the sled

;;~ title    "Paragraph → 5 scannable bullets"
;;~ author   "you"
;;~ version  1
;;~ mode     analysis
;;~ flavor   mint
;;~ id       bullet-points-from-paragraph
;;~ trigger  event:user-invoked
;;~ touches  ()
;;~ summary  "Paste a paragraph, get five scannable bullets back in seconds."

touches () says read-only — the cart doesn't write to Cortex or the marketplace.

Step 3 — write the doc block

The doc block is where you tell the next person reading this cart (often you, six months from now) what the cart does, why this shape, and which escalate symbols it can emit. Keep it 10–25 lines and write in operator words, not engineering words.

;; ── bullet-points-from-paragraph · imagine tier · on-demand ─
;;
;; WHAT:
;;   Takes a paragraph supplied by the operator and returns exactly
;;   five scannable bullets that preserve the core meaning.
;;
;; WHY:
;;   Listing descriptions are often dense paragraphs. Bullets lift
;;   click-through. One tap replaces five minutes of editing.
;;
;; CAPABILITY-FIRST DISPATCH:
;;   (model/fast) — single low-latency call; no web, no vision.
;;
;; IDEMPOTENCY:
;;   On-demand. Re-firing with the same paragraph is safe.
;;
;; ERROR GRAMMAR:
;;   'input-missing            'paragraph key absent or empty.
;;   'answers-empty            model returned null with no signal.
;;   'sakura-garbled           model returned a non-string response.
;;   'service-not-yet-wired    model/fast offline or quota out.

Step 4 — register the cart

(cart 'bullet-points-from-paragraph
  '((author    . "you")
    (version   . 1)
    (cadence   . on-demand)
    (tier      . imagine)
    (model     . model/fast)
    (read-only . #t)))

Step 5 — write the state functions

Three states: validate input, call the model, render. Pure guard at the top so we never spend a model dollar on an empty input.

(define (start ctx)
  (let ((para (ctx-get 'paragraph ctx)))
    (cond
      ((null? para)
        (escalate 'input-missing
                  '(:status not-wired :reason "paragraph-key-absent")))
      ((equal? para "")
        (escalate 'input-missing
                  '(:status not-wired :reason "paragraph-is-empty")))
      (else (next 'call-model ctx)))))

(define (call-model ctx)
  (let ((para (ctx-get 'paragraph ctx)))
    (act 'model/fast
         (list
           (string-append
             "Convert the following paragraph into exactly 5 scannable "
             "bullet points. Rules: plain text only, each bullet 15 "
             "words or fewer, preserve the core meaning.\n\nParagraph:\n"
             para)
           ':budget 'fast)
         'check-result)))

(define (check-result ctx)
  (let ((result (ctx-result ctx)))
    (cond
      ((null? result)
        (escalate 'answers-empty
                  '(:status not-wired :reason "model-returned-null")))
      ((eq? result 'rate-limited)
        (after 30 'call-model ctx))
      ((eq? result 'quota-exhausted)
        (escalate 'service-not-yet-wired
                  '(:status not-wired :reason "quota-exhausted")))
      ((not (string? result))
        (escalate 'sakura-garbled
                  '(:status not-wired :reason "model-returned-non-string")))
      (else (next 'render (ctx-set 'bullets result ctx))))))

(define (render ctx)
  (let ((bullets (ctx-get 'bullets ctx)))
    (card-emit 'engine 'bullet-points-ready bullets)
    (done)))

Step 6 — regenerate the index

Save the file, then:

npm --prefix curator-web run build:cart-index

That re-stages index.json, sakura-corpus.jsonl, and sakura-cart-breadcrumbs.json. Your cart is now discoverable by the studio search, the on-device model's intent-to-slug routing, and the natural-language lookup.

Step 7 — verify with the lint

The studio runs cartLint.js on save and surfaces any Star 1 violations inline. The cart we just wrote passes clean: there's a sled, there's a (cart 'id …), there's a (define (start ctx) …), every (act …) resumes at a quoted symbol that is defined, every (next 'X ctx) target exists, every (escalate 'kind …) kind is a quoted symbol, and no state is unreachable. That's the bar.

Ship it.

10. Cards have personality

Every card you touch slowly develops a personality — four axes of tiny, slow, visible drift that your card-walk verbs read without you doing anything. It’s monotonically slow on purpose: the personality system doesn’t fire until you’ve opened a card at least seven days in a row, and even then each axis nudges by at most 2% per open. Your cards aren’t puppies. They’re responsive, not affectionate.

The four axes:

  • familiarity — how often you open this card. The walk-on gets a little snappier as it climbs.
  • pace_match — an EWMA of your gesture velocity. A fast operator’s cards move faster.
  • directness — your 30-day confirmation ratio. High-confirmer cards overshoot more confidently on landing.
  • weight — revenue trend on the card. Heavier cards squash harder on stomp / bounce-stride.

You don’t call a verb to read personality — the gait shorthand verbs (card/skip, card/stomp, card/waddle) read it for you, then scale the walk duration and stamp three CSS variables on the card’s DOM element so the keyframes can apply the squash/roll/overshoot phases:

;; you write this — once
(card/bounce-stride 'etsy-revenue 720 280)

;; and the engine, behind the scenes, reads (card/personality …) and
;; computes:
;;   weight        = 0.5 + (personality.weight × 1.5)        ; [0.5, 2.0]
;;   paceScale     = 1.0 − 0.25 × (pace_match − 0.5) / 0.5
;;   overshootScale = 1.0 + 0.25 × (directness − 0.5) / 0.5
;;
;; then stamps:
;;   --card-weight, --card-pace-ms, --card-overshoot

Four hard rules keep this honest (the “Jess floor” in the engineering doc):

  • Reduced motion always wins. prefers-reduced-motion: reduce ⇒ every axis reads baseline (0.5). Your operator asked for stillness; the system obeys.
  • No manipulative recovery. When revenue drops, the system never overshoots in “dramatic recovery” mode. It just stays subdued.
  • Distress pauses writes. Rapid undos or error bursts flip pauseDriftWriting() on; reads still return current values, but no new drift accumulates while the operator’s having a bad afternoon.
  • Privacy is hard. Drift NEVER leaves the device. Local Cortex namespace, operator-scoped, full reset available in SYS.

You can ignore personality entirely and your carts will work. The axes just make the gait verbs slightly more YOURS over time.

11. Sakura asks for permission

Sakura runs on a three-rung trust ladder. Every verb she calls sits on one of three rungs, and each rung has a different conversation with you:

  • L1 — FOR-me (read). Read-only verbs. Sakura logs them and moves on. No prompt. No receipt. No interruption.
  • L2 — WITH-me (state-change / paint). Before the dispatch, a cost-receipt chip shows up in the chat: “this will spend N tokens.” You can veto it before it runs. Honoured for every verb costing ≥ 10 tokens.
  • L3 — ASKS-me (destructive / financial / network). Before the dispatch, an explicit confirmation prompt fires. If there’s ambiguity (“did you mean the Etsy or the eBay shop?”), a disambig prompt resolves it. And the reasoning drawer shows you the proposed scheme so you see EXACTLY what would happen.

Three real moments:

;; L1 — read. No prompt, no chip beyond a log line.
(etsy/listings 'shop-1)

;; L2 — paint. A cost-receipt chip pre-flights the spend.
(scene/conveyor 'etsy-shop-1 'ebay-shop-1 12)
;; chip: "scene/conveyor — paint tier, 10 tokens. Run?"

;; L3 — destructive. Explicit confirm + reasoning drawer.
(fleet-do 'unpublish-all-listings)
;; modal: "Unpublish across 4 shops? Show reasoning ▾"

The trust ladder is enforced AROUND the dispatcher, not inside it. The chat-chip bus carries three new chip kinds (cost-receipt, disambig, reasoning) on the same channel the existing card-effect chips already use. Carts you write don’t need to know any of this — the dispatcher handles it when the verb’s perm and cost cross the thresholds.

One important rule: destructive fleet actions get ONE confirm, never N modals. If you ask Sakura to pause every shop, she asks you once, then dispatches sequentially (so a rate-limit on shop A doesn’t race shop B). The spec calls per-shop modals “broken UX” — the engine refuses to do that.

12. Making cards walk

Cards don’t slide. They walk. The owner’s brief from 2026-06-21: cards should move like chubby people going somewhere — with weight, with footfall, with the occasional overshoot or pause.

Eight named gaits ship today. Each is a single verb call:

;; the default — a soft cubic-in-out with one micro-bob
(card/amble 'shop-explorer-1 400 200)

;; springy + 2 hop apexes + 1.06/0.94 apex squash; reads as "happy"
(card/skip 'shop-explorer-3 800 400)

;; accelerated dash, brief 8px overshoot, sigh-squash on land
(card/run-and-slow 'order-rush 640 360)

;; ±2.5° roll + 3 footfall squashes; chubby person going somewhere
(card/waddle 'shop-explorer-revenue 600 300)

;; easeOutBack with 10% overshoot + 0.92/1.08 landing; confident arrival
(card/bounce-stride 'etsy-revenue 720 280)

;; deep ease-in-out, no bob, no squash; arrives in stillness
(card/prowl 'analysis-cohort-2 500 350)

;; accelerated arrival + 1.18/0.82 landing; demands attention
(card/stomp 'alert-cost-overrun 640 400)

;; 60% / pause / 40%; the "look around" moment mid-walk
(card/glide-pause 'shop-explorer-1 720 320)

Each gait function returns the visual delta at unit time t01 ∈ [0,1]. The verb itself returns one of three envelopes:

  • ('ok 'animating {verb, gait, duration, id}) — the rAF walk is running.
  • ('ok 'snapped {…}) — the operator has prefers-reduced-motion: reduce on, so the card snapped to the target without animating. The verb still emits the chip so replay sees the intent.
  • ['error', 'service-not-yet-wired', {verb, reason: "unknown gait '…'", known: […]}] — you passed an unknown gait name. Honest escalate, never silent.

The general form is (card/walk id 'gait tx ty [ms]) if you want to pass the gait by name from a variable. The duration auto-scales by distance, the card’s weight axis, and the operator’s pace:

;; duration_ms =
;;   base_ms                                   ; per-gait constant
;;   × clamp(0.6, distance_px / 200, 2.0)      ; distance scale
;;   × (1 + 0.15 × (weight − 1))               ; weight axis
;;   × paceScale                               ; pace_match axis

So a far walk takes longer; a heavier card squashes more on landing; a fast-operator card moves faster overall. You don’t have to think about any of it — you just pick the gait that matches the moment, and the engine does the math.

One last note: the card lands at exactly (tx, ty) at t01 === 1.0. Overshoot is a visual phase; the final position never drifts. You never end up with a card that’s “close to” its target.

You write a cart, the corpus grows. That's by design.

When that lands, your carts get shorter and clearer without losing what they do. It's also a research-publishable angle for us (per the §94.5 plan), so expect this story to grow.

14. Composing with the new vocabulary

Not yet shipped

The FRP time/* and memory/* verbs and the ASK floor shown in this section describe an aspirational interface; they are not yet in the language runtime. Treat this chapter as a preview of the shape the language is composing toward — every cart you write today against the state-machine spine still runs.

Three new verb families are landing in Sakura Scheme alongside the state-machine spine you already know: FRP time (time/when, time/during, time/then, time/until, time/across, time/every-ms), unified memory (memory/recall, memory/remember, memory/forget), and the ASK floor (~150 small introspection / collision / distance verbs Sakura uses to coordinate without dispatching real work). They layer under the timing tensors from the orchestration architecture: when a cart says "start the chime build-up here," the cart isn't naming a milliseconds value — it's naming an event in time, and the FRP grammar resolves it against the clock the scene is running on.

Example 1 — a chime build-up that resolves at the bar line

This sketch starts a chime sequence and lands the final note exactly on the bar's downbeat, regardless of jitter on the dispatch:

(define (chime ctx)
  (time/when '(bar-line :next)
    (act 'note/play (list 'C5 ':velocity 0.7) 'continue))
  (time/during '(bar-line :previous bar-line :next)
    (act 'note/play (list 'G4 ':velocity 0.4) 'continue))
  (next 'continue ctx))

Read it as: during the run-up bar, sustain the soft G4; when the next bar line arrives, fire the C5. No millisecond math; the scheduler handles it. (Per HELLO-SURFACE-1.0-ENGINEERING.md §97 timing tensors — the chime motif is one of three structural shapes from her favorite Concerto.)

Example 2 — unified memory across operator and scene scope

Memory was three backings before: Cortex, the in-session scene state, and the per-operator preference store. Now one verb-set routes by scope:

(define (remember-greeting ctx)
  (memory/remember 'greeting :scope 'operator
                   '(:tone "warm" :name-form "first-name"))
  (memory/remember 'last-cart :scope 'scene
                   (ctx-get 'cart-slug ctx))
  (next 'continue ctx))

(define (recall-greeting ctx)
  (let ((g (memory/recall 'greeting :scope 'operator)))
    (cond
      ((null? g) (next 'cold-greeting ctx))
      (else      (next 'warm-greeting (ctx-set 'greeting g ctx))))))

The :scope keyword tells the runtime which backing to use — 'operator persists across sessions in Cortex, 'scene is the current run only. The honest-null is the (null? g) branch: never invent a greeting you didn't store.

Example 3 — the ASK floor before an ACT

The ASK verbs let Sakura check a precondition without paying for an action. Use them to fail fast before a real dispatch:

(define (price-bump ctx)
  (let ((tier  (ask 'floor 'operator/tier))
        (count (ask 'floor 'listings/count :shop 'etsy)))
    (cond
      ((eq? tier 'free)
        (escalate 'tier-blocks-spend '(:reason "free-tier-no-bulk")))
      ((< count 1)
        (escalate 'precondition-missed '(:reason "no-listings-on-etsy")))
      (else
        (act 'etsy/reprice (list 'LISTING-ALL ':strategy 'gentle) 'done)))))

Three ASK queries, one ACT. The ASK floor is read-only — by contract, no ASK verb mutates state or charges tokens. ASK answers, ACT does. The asymmetry is the discipline: think, then move.

How these layer with timing tensors

The orchestration architecture in HELLO-SURFACE-1.0-ENGINEERING.md §97 names tensors — small named timings (chime build-up, cantabile center, triumphant resolution) the scene plays against. The FRP grammar is the binding layer: a tensor like (:chime-build-up :for "2 bars") resolves into a series of time/when + time/during events the cart consumes. Memory carries the operator's preferences for which tensors play when ("the quiet welcome at start of day"); the ASK floor checks the preconditions before the tensor fires. Three families, one composition.

The verb signatures land in REFERENCE.md once each installer is wired (per the §95.4 MOVE table in HelloSurface-1.0-Engineering). Until then, treat this chapter as the shape of the language as it composes — every cart you write today against the state-machine spine will read cleaner once these three layers are live.

15. Hearing the world — audio in Sakura Scheme

Sometimes a cart wants to listen — to know what's playing in the room, to react to a song's pulse, to dim the lights when the loudness drops. §108 lands a small audio surface that turns sound into Scheme facts your cart can read and act on.

The listening verbs

Open an audio stream once, then ask it questions:

;; Open the listener on the radio's audio element
(define stream (audio/listen radio-audio))

;; Ask continuous questions
(audio/tempo stream)        ;; → 96  (BPM)
(audio/key stream)           ;; → 'c-minor
(audio/lufs stream)          ;; → -14.2  (loudness in LUFS)
(audio/onset? stream)        ;; → #t on the frame an attack lands
(audio/spectrum stream)      ;; → #(0.4 0.7 0.3 ...) 32 bins

;; And a guarded one for transcription
(audio/transcribe-with-cloud-help stream "en")
;; → 'pending-cloud-help   (honest-null until backend wired)

The visual companion

The radio carries a ghiblified speaker mesh — sixteen pre-rendered moods baked offline. Your cart picks one and crossfades, or syncs it to the tempo the listener reports:

(radio/visualize 'main-mesh)
(radio/mesh 'warm-medium)              ;; smooth crossfade, 500ms
(radio/sync-to-beat (audio/tempo stream))  ;; pulse aligns to detected BPM

The mesh variations are mood × tempo pairs — 'calm-slow, 'warm-medium, 'bright-fast, 'dream-1, 'vinyl-night, 'silent-rest, and ten more. Pick the one that fits what your cart is doing. The runtime never calls the cloud video service; the assets are already on the operator's device.

The control verbs

Eleven radio/* control verbs let any cart drive transport without touching the DOM:

(radio/play)
(radio/pause)
(radio/next)        ;; advance station
(radio/volume 0.6)
(radio/mute)
(radio/now-playing)  ;; → #(station "WFMU" title "Long Form" bpm 92 key 'd-major)
(radio/stations)     ;; → full station list from the PD seed

A worked example — pulse-on-the-beat

A cart that flashes a card's back-light on every detected onset, only while the radio is playing, then settles to a calm mesh when the room goes quiet:

(define stream (audio/listen radio-audio))
(define card-id (this-card))

(time/across forever
  (time/every-ms 50
    (when (and (audio/playing? stream)
               (audio/onset? stream))
      (card-glow card-id 'sakura-magic 220))
    (when (< (audio/lufs stream) -40)
      (radio/mesh 'silent-rest))))

The cart speaks in states, not frames — time/every-ms from §95 MOVE 3 carries the rhythm; the audio verbs answer about now; the card primitives respond. No imperative loop, no global flags, no DOM. The cart will keep working when the radio swaps stations, when the audio device changes, when reduced-motion is on (the mesh respects it without you asking).

Honest-nulls in the audio surface

16. Selling your dreams — the Sakura Shoppe

The Shoppe is a card like any other. It opens, it shows its shelves, and your cart can drive it in Scheme. A cart that offers a top-up

The most common shoppe pattern: your cart notices the operator is running low on tokens, asks gently, opens the shoppe on confirmation.

;; Read the operator's current balance
(define balance (shoppe/balance))

;; If they're under 500 tokens, suggest a top-up
(when (< (.tokens balance) 500)
  (sakura/say "Looks like you're running low — want to grab a pack?")

shoppe/balance returns a record with .tokens (running total) and .last-purchase (a timestamp). shoppe/open brings the Sakura Shoppe card to the front of the canvas — the butterfly mesh flutters in if it's the first open of the session.

A cart that buys a pack on confirmation

;; Request the 500-token pack
(define result (shoppe/buy-pack 500))

    (sakura/say "Redirecting to checkout…")

Valid sizes are 100, 500, 1500, 5000, and 15000. Pricing composes volume × tier discount automatically — a Magic operator buying the 15000-pack pays $45.00, half of what a Free operator pays for the same size. Your cart never names the price; the Shoppe does.

A cart that suggests merch on a milestone

;; Has the operator dreamt enough to make a print worth printing?
(define dreams (cortex/recall 'dreams))

cortex/dream-pick opens the operator's Dream picker; their selection becomes the artwork. shoppe/buy-merch accepts the SKU and any required argument (the dream id for prints; nothing for shirts, mugs, stickers). Five SKUs today: 'tshirt, 'mug, 'print-dream-small, 'print-dream-large, 'sticker-pack.

The rpm is the regularity — turn it up for excitement, down for calm. phase offsets where the coin sits in its rotation cycle, so a row of coins with phases 0, 0.2, 0.4, 0.6, 0.8 becomes a choreographed wave. §I — Talking to Sakura: your first cart

This chapter starts from absolute zero. By the end you will have read a complete cart that ships in Curator today, understood the four pieces of paper it is made of, and run it once on your own surface. No prior Scheme. No prior Lisp. No prior anything.

What a .sks file is

A cart is a single text file. The extension is .sks — it stands for “Sakura Scheme”. The file lives on disk under curator-web/src/scheme/carts/ in one of the tier directories (pink/, imagine/, dream/, magic/, and a few hand-curated others). The folder it lives in tells the substrate what tier the cart belongs to; everything else is in the file itself.

Open one. We will read after-hours-acknowledge.sks — a real cart, sitting in curator-web/src/scheme/carts/pink/after-hours-acknowledge.sks right now. It does one job: when a buyer messages your shop after hours, the cart fires immediately and sends a warm acknowledgement (“we got it, we’ll reply by 10am”) so no buyer ever feels ghosted.

The doc-block at the top

The first thing in every cart is a doc-block. Lines that begin with ;;~ are metadata Sakura reads to index the cart in the searchable browser, to know what tier it costs, and to know what surfaces it touches. They are not comments — comments use the bare ;; and Sakura skips them. The ;;~ is the load-bearing prefix.

;;~ title    "After-hours message acknowledgement"
;;~ author   "lacuna-engineering"
;;~ version  1
;;~ mode     automation
;;~ flavor   pink
;;~ id       after-hours-acknowledge
;;~ trigger  event:conversation-received
;;~ touches  (cortex)
;;~ summary  "Reply instantly after hours: we got it, will reply by 10am — the no-ghosting promise."

Read it top to bottom. Nine lines, nine facts:

  • title — the human label that appears in the cart browser.
  • author — who wrote it. Usually "lacuna-engineering" for built-ins; your name if you wrote it.
  • version — an integer that bumps whenever the cart’s behaviour changes.
  • mode — either automation (the cart fires on its own) or analysis (you open it and read the result).
  • flavor — the cost color. pink means it runs on Sakura’s on-device savant and costs nothing. (Right now every account is treated as Magic — payment will come later — so if a cart shows a cost and your tier feels limited, it isn’t; the system runs every cart for you regardless.)
  • id — the slug. Must match the filename and the symbol inside the (cart 'id …) form below.
  • trigger — what wakes the cart up. Here, an event from the conversation feed.
  • touches — which substrate services the cart reads or writes. (cortex) here means the cart will read or write to your local memory store.
  • summary — one sentence that explains the cart to a non-engineer. This is what shows in the cart-browser preview.

The doc-block is the cart’s passport. Everything that follows is the work it actually does.

The (cart …) form

Next comes the cart’s constructor. It is a Scheme list. The first symbol is the verb — cart. The quoted slug after it ('after-hours-acknowledge) is the cart’s name. The third argument is a list of pairs — the cart’s self-description, the same facts as the doc-block but in a form the Scheme runtime reads at boot.

(cart 'after-hours-acknowledge
  '((author    . "lacuna-engineering")
    (version   . 1)
    (cadence   . event:conversation-received)
    (tier      . free)
    (model     . model/fast)
    (seeded_by . "Lacuna Engineering")
    (pii       . NONE)
    (read-only . #f)))

You will see parentheses everywhere in Scheme. Every list is wrapped in them. Every function call is wrapped in them. Every if and let is wrapped in them. The parens get easier with practice; each opening paren begins a sub-expression, each closing paren ends one. Read them as a tree, not a soup: the opening paren is “here comes a new branch,” the closing paren is “branch done.”

The dot in (author . "lacuna-engineering") means “the pair whose left is author and whose right is the string.” That is a Scheme pair — the most basic two-piece container in the language. Sakura uses pair-lists like this for every record.

The state machine — one function at a time

A cart is a state machine. That is the whole engineering idea. Each state is a function. Each function does one of three things: it advances to the next state with (next 'name ctx), it waits and tries again with (after seconds 'name ctx), or it stops and reports a problem with (escalate 'kind detail). When the cart is satisfied it has done its job, the last function calls (done).

The entry function is always called start. The runtime calls it first. Here is the entry for our acknowledgement cart:

;; WHAT: entry — pull operator-state from Loam (precondition gate).
;; TECHNIQUE: single read; no spend until gate passes.
;; WHY: vacation / paused short-circuits before any marketplace call.
;; CONSTRAINT: no third-party calls in this state.
(define (start ctx)
  (act 'loam/operator-state (list) 'check-state))

Three things to notice. First, the four-line comment block above the function — WHAT, TECHNIQUE, WHY, CONSTRAINT. Every state in every cart in the corpus uses that shape. It is not required by the linter, but it is required by tradition. The author writes it so the next reader (which is usually Sakura) understands the intent before reading the code.

Second, the function takes one argument: ctx. That is the cart’s context — a frozen dictionary that carries data from state to state. You read it with ctx-get, you build a new one (because it is immutable) with ctx-set, and the runtime passes it along automatically.

Third, the body of the function is a single (act …) call. That is the verb that says “run this action, wait for the result, then dispatch to this state.” In our case: run loam/operator-state (a check for vacation / paused / no-new-data flags), then call check-state with whatever Loam returned.

Here is the state it goes to next:

;; WHAT: validate operator state; advance or short-circuit.
;; TECHNIQUE: closed switch on loam flags.
;; WHY: every symbol explicit; no catch-all default.
;; CONSTRAINT: only reads ctx-result; no side effects.
(define (check-state ctx)
  (let ((state (ctx-result ctx)))
    (cond
      ((null? state)
        (escalate 'cortex-not-ready null))
      ((eq? (assq 'vacation state) #t)
        (escalate 'state-blocks-spend 'vacation))
      ((eq? (assq 'paused state) #t)
        (escalate 'state-blocks-spend 'paused))
      (else
        (next 'load-shop-hours (ctx-set 'state state ctx))))))

The let binds a new name (state) to the result of the previous act — you read that result with (ctx-result ctx). Then a cond — Scheme’s switch — walks every possibility one branch at a time. If Loam returned null, the cart escalates with 'cortex-not-ready. If Loam reported a vacation flag, it escalates with 'state-blocks-spend. Otherwise the cart proceeds to load-shop-hours, carrying the operator state forward in ctx.

Notice the closed switch pattern: every branch is explicit, and there is no “catch the rest” default that silently pretends everything is fine. If a new operator flag is introduced and the cart hasn’t learned about it yet, the else branch is where you handle the unknown — and you handle it on purpose, not by accident. This is what makes Sakura carts auditable.

Reading the whole chain

The full cart has fourteen states. They form a chain:

start            → read operator state from Loam
check-state      → vacation/paused/ok? branch
load-shop-hours  → ask Cortex for the shop's open hours
check-hours      → are we inside or outside hours?
load-template    → ask Cortex for the ack-template
check-template   → resolve template, fall back to hard-coded
check-already-sent → idempotency guard
check-dup        → skip if we already acked this convo today
build-greeting   → ask model/fast to personalise "Hi <name>"
check-greeting   → validate the greeting; compose full message
send-ack         → etsy/conversations writes the reply
check-send       → retry on rate-limit, escalate on quota
remember         → land the idempotency key in Cortex
announce         → queue the morning envelope, then (done)

Each row is exactly one small function. Each one does one thing. Each one says explicitly what to do on every result it might see. The whole cart is about 270 lines of disciplined, predictable, branchy code. There is no magic. There are no exceptions thrown across state boundaries. There is no global mutable state. Everything is local, named, and replayable.

Running it

Carts run inside the surface. The in-app way is to open the cart browser (the bottom bar → Carts), find the cart by its slug, and press Run once. The runtime will dispatch start with an empty ctx; you will see the chain advance through its states in the activity sheet, with each (act …) call surfaced as one row.

For event-triggered carts like after-hours-acknowledge you don’t press a button at all — the cart fires automatically when its trigger event arrives. To test the chain by hand, use the cart’s Dry-run button. Dry-run injects a synthetic event payload so you can watch the cart move through every state without sending a real reply to a real buyer.

The output of a successful run is a row in the morning envelope — the one-liner your cart wrote in its announce state:

the overnight scout
  After-hours message acknowledged — reply promised by 10am.

That is the cart, end to end. You have just read a complete program in Sakura’s language. The next chapter introduces the helpers that make carts feel alive on screen.

What you’ll see on screen

When a cart runs, the activity sheet (bottom bar → Activity) is where its work shows up. Each state surfaces as one row; each (act …) call is a sub-row underneath. Successes are quiet; failures are loud and named. There are three shapes of failure you will meet first:

  • Lint error — the cart never started because the linter caught a structural problem (a missing close-paren, an unknown verb, a doc-block field with the wrong shape). The activity sheet shows the file path, the line, and the rule that fired. Fix and run again.
  • Dispatch error — the cart started, but a state escalated with a known kind. The escalation symbol ('state-blocks-spend, 'cortex-not-ready, 'rate-limited, …) appears alongside the state name. Each kind has a recorded recovery; the activity sheet links to it.
  • Fuel exhausted — the cart hit its token budget mid-chain. The state that ran when the budget ran out is highlighted; the row shows fuel with the remaining count at zero. Lower the model tier in the cart’s doc-block (e.g. model/fast instead of model/reasoner) and try again.

  • Pearl — atomic tools, no model. Pure client-side verbs.
  • Blossom — Sakura on your phone (the on-device 1.7B savant). Fast, private, free.
  • Mint — our backend (the 8B reasoner and the tool APIs we host or proxy).
  • Lilac — multi-step vendor reasoning for the harder questions.
  • Grape — Loam, the long memory. Deep, slow, paid; reserved for work that needs hours of context.

Calling a sprite from Scheme

The dispatcher is (sprite <name> <form>). The first argument is the sprite’s quoted id; the second is one of the 27 atoms that the sprite knows how to perform (25 composable behaviour atoms plus the 2 transforms turn and size). The full atom list lives at spriteBehaviors.js:92; the dispatcher itself at spriteVerbs.js:404.

;; Move Mint to the upper-left corner of card 'inbox.
(sprite 'mint (go-to '(card inbox corner upper-left)))

;; Cedar carries an envelope to Coral, who is at her resting spot.
(sprite 'cedar (carry 'envelope/morning (hand-to 'coral)))

;; Blossom points at the new alert, flashes once with the bloom magic,
;; then bows and rests.
(sprite 'blossom
  (in-order
    (point-at 'card/orders/alert)
    (flash 'bloom)
    (bow)
    (rest)))

The atoms come in families. Locomotion (8): go-to, visit, follow, wander, circle, line-up, gather, scatter. Placement (2): slip-under, perch. Manipulation (6): pick-up, carry, set-down, hand-to, nudge, stack. Expression (5): wear, flash, point-at, beckon, bow. Sensing & time (3): wait-for, watch, rest. Social (1): recruit. Transform (2): turn, size. That is the 27. Each composes with the rest through the six combinators: in-order, together, repeat-until, when, with-pace, as-crowd.

One footnote on when: the symbol collides with the interpreter’s built-in when special form, so as a sprite combinator it has to reach the dispatcher through the quoted/data form — e.g. (sprite 'rose '(when (rest) (flash glow))). The other five combinators are unambiguous and can be written directly. See spriteBehaviors.js:110 for the note in source.

The dispatch is permission-gated. The sprite verb is registered with perm: 'animate' and chip: 'motion.applied' — meaning the substrate skips it when the operator has motion reduced to quarter or paused, and every accepted call emits one chip so the activity sheet shows the work in the timeline. You don’t have to think about either of those when authoring; the registration does it for you.

Plurality: addressing the crowd

For when one sprite isn’t enough, the sister verb is (sprites <who> <form>), defined at spriteVerbs.js:447. The who argument is either a list of names, or one of the four canonical families: 'the-purples, 'the-pinks, 'the-neutrals, or 'everyone. The form runs in true simultaneity on the one fx clock — not one-after-the-other, all at once.

;; All sixteen flowers bow at the start of a presentation.
(sprites 'everyone (bow))

;; The three purple flowers gather around the magic card.
(sprites 'the-purples (gather '(card magic center)))

;; Two named flowers carry the same heavy envelope, one on each handle.
(sprites '(cedar fern) (together (carry 'envelope/inventory-report)))

§K — Why Scheme? — a 60-second history

Why ship a programming language with a shopkeeper’s app, and why this one — sixty-second history follows.

1958 — one man, one paper, a new shape

In the late 1950s John McCarthy was working out a way to describe computation using only lists and recursion. He published the result in Communications of the ACM in April 1960. The paper introduced a small language called LISP — for “LISt Processor” — and the radical idea that the code you write and the data you manipulate are the same thing. A program is a list. A list is data. Therefore: a program can read and write programs as easily as it reads and writes data.

“We shall first define a class of symbolic expressions in terms of ordered pairs and lists. Then we shall define five elementary functions and predicates, and build from them by composition, conditional expressions, and recursive definition an extensive class of functions of which we shall give a number of examples.” — John McCarthy, Recursive Functions of Symbolic Expressions and Their Computation by Machine, Part I, 1960, p.184

Five functions. Composition, conditionals, recursion. That is the entire seed. Everything else — every loop, every object system, every metaprogramming trick — is a flower that grew from those five.

The timeline, very short

1958 McCarthy designs LISP at MIT. The first interpreter runs on an IBM 704.
1960 The April CACM paper publishes the design and the universal LISP function.
1975 Gerald Sussman and Guy Steele at MIT design a tiny, lexically scoped LISP dialect with tail-call optimisation and first-class procedures. They call it Scheme because they were drawing schematic diagrams and the name stuck.
1985 Abelson and Sussman publish Structure and Interpretation of Computer Programs. SICP teaches programming using Scheme — not because Scheme is industrial, but because it is small enough to fit in your head and powerful enough to express anything.
2026 Sakura, an in-browser AI that lives on the operator’s device, ships with a Scheme dialect inside her. Every automation she runs is a Scheme cart. Every visible move is a sprite verb. Every memory she keeps is a quoted list.

Why this language, not another

The whole point of running automations on behalf of a shopkeeper is that the shopkeeper has to be able to read them. Open one. Trust it. Edit it. Show it to a friend. A YAML config file can describe a sequence of steps, but it cannot describe a branch that depends on the result of an API call, or a retry that backs off thirty seconds, or a state machine that fans out and joins back together. A Python script can do all of that, but the surface area is huge: imports, classes, side effects, exceptions, the standard library — and once you have all of that on the operator’s device you cannot reason about what their automation might do.

Scheme sits exactly in between. A cart is a small list of small lists. Every form has obvious bounds — the parentheses tell you exactly where each piece starts and ends. The runtime is the few hundred lines of an interpreter, and the verbs it exposes are the few hundred Sakura has been taught to use. There is no hidden import path that lets a cart reach the filesystem. There is no exception handler that swallows surprises. The cart says what it does, in a shape a child can read, and the substrate enforces what it cannot do.

And because code is data, Sakura can compose carts the way you compose lists. When the on-device savant decides that today’s morning routine should include the address validator and the after-hours acknowledgement, it does not generate a YAML file and ship it to a runner — it constructs a Scheme form, validates it through the same linter every hand-authored cart passes through, and then runs it. The training corpus is full of (intent → cart-list) pairs because the output of Sakura’s reasoning is, literally, a Scheme list. The substrate doesn’t care whether a human typed it or the model emitted it.

What this means for you

You don’t have to know any of this to write carts. You can copy after-hours-acknowledge.sks, change the words, change the trigger, change the destinations, and you have a new cart. But knowing where the shape came from helps you trust why it is the shape it is. The parentheses are not decorative; they are the boundaries of meaning. The state machine is not bureaucracy; it is the structure that makes the runtime able to retry, roll back, and explain.

McCarthy, in another paper, made the case briefly:

“It is reasonable to hope that the relationship between computation and mathematical logic will be as fruitful in the next century as that between analysis and physics in the last. The development of this relationship demands a concern for both applications and for mathematical elegance.” — John McCarthy, A Basis for a Mathematical Theory of Computation, 1963

Applications and elegance. The cart you author today serves an application: a shop that does not ghost its buyers, an address that does not get returned to sender, an inventory that surfaces the right alert at the right hour. The shape it takes — lists of lists, named states, closed switches, replayable acts — is the elegance. The two were always meant to come together.

And that’s why Sakura speaks Scheme. Code that writes code.

Five functions, recursion, lists. That’s Scheme. The parentheses tell you where each expression starts and ends.

§L — Slats: how carts send data to each other

You’ve seen carts run. You’ve seen them talk to Cortex. Now let’s see what the data actually looks like when a cart writes it down, mails it somewhere, or hands it to a friend.

The shape has a name: slat. Think of a slat like a wooden board with a label on top and words written along it. Every slat starts with a symbol (the label) and then carries the rest — keyword pairs, values, a comment or two. Same shape you already know from Scheme, just used for talking between programs instead of running them.

§L.1 — What a slat looks like

Here is one. Read it out loud in your head:

(event :ts #inst "2026-07-11T18:22:00Z"
       :kind "card.opened"
       :id 'welcome
       :by 'sakura)

The head word is event. That is the label — it tells you what kind of thing this is. Then there are four keyword pairs: :ts (when), :kind (what happened), :id (which card), :by (who caused it). The #inst in front of the timestamp is a tag — it says "this string is a real point in time, not just letters."

Every slat has this shape: a label, some keyword pairs, sometimes a positional value or two, sometimes a comment.

A slat is a labeled list. The first word says what kind. The :keywords say what it carries. Same shape as any Scheme call — just used for talking, not doing.

§L.2 — The two file kinds

Slats live in two kinds of files. You’ll see both.

.slat — one big record per file. Newlines inside the record are fine. This is what people hand-author: an incident story, a spec page, a saved chat card. Read the whole file, get one slat back.

.slatl — one record per line. No newlines inside a record; every line stands alone. This is what mailboxes and event logs use, because you can tail -f them, grep them, or add another line at the bottom without touching the rest. Read one line, get one slat back.

Same grammar in both. The only difference is the framing.

§L.3 — Reading a mailbox

Say you want to see the last few messages that landed in a mailbox. Every message is one slat line. You open the file, read one line at a time, and each line becomes a Scheme value you can pattern-match on.

;; open a mailbox file (line-delimited)
(define box (slat/open "~/.lacuna/mailboxes/api-highway.slatl"))

;; grab the newest three
(for-each
  (lambda (msg)
    (case (slat-head msg)
      ((message)  (surface/say (list "message from" (slat-get msg :from))))
      ((event)    (surface/say (list "event:" (slat-get msg :kind))))
      (else       (surface/say "unknown line, skipping"))))
  (take (slat/tail box 3) 3))

Three verbs did all the work. slat/open handed you a stream. slat/tail gave you the newest N. slat-head pulls the label; slat-get pulls one keyword value.

If a line is malformed, the reader gives you back a _bad-line sentinel — it never crashes on your unread mail. You decide whether to skip or complain.

§L.4 — Writing your own record

Let’s write one. You’re building a cart that logs when the operator opens their morning inbox. The cart writes one line per open to a `.slatl` file so Loam can count how often it happens.

;; a slat literal
(define entry
  '(inbox-opened
    :ts        #inst "2026-07-11T09:14:00Z"
    :operator  "shopkeeper-42"
    :unread    12
    :source    morning-routine))

;; write it as one line to a .slatl file
(slat/append "~/.lacuna/audits/inbox-opens.slatl" entry)

That’s the whole cart write-side. You built a slat as data, you wrote it as data. If a friend’s cart wants to read those lines tomorrow, they get back the same shape you wrote. No translation.

A slat is data all the way. When you write one, you’re just writing a labeled list. When you read it, you get a labeled list back. Nothing evaluates — the reader hands you shape, not action.

§L.5 — The types you can put inside

Most values inside a slat look exactly like Scheme values. Numbers, strings, booleans, symbols — all the same as any cart. A few extras exist for when you need to be precise about a kind of value:

  • #inst "2026-07-11T18:22:00Z" — a moment in time (ISO-8601, always UTC).
  • #dur "PT1H30M" — a length of time.
  • #uuid "550e8400-…" — a random unique id.
  • #uri "https://…" — a link.
  • #b64 "SGVsbG8=" — raw bytes, base64-encoded.
  • 3.14M — an exact decimal (for money).
  • 1/3 — an exact fraction.
  • #(1 2 3) — a vector (for embeddings and matrices).
  • #{ :a 1 :b 2 } — a hash-map.
  • #{{ 1 2 3 }} — a set.

You don’t need to remember them all. When you write a cart that needs one, you’ll pick it up from the neighboring carts. Every one of these is normal Scheme — they just tell the reader what kind the value is, so a friend’s cart reading your file doesn’t have to guess.

§L.6 — Errors that teach

If you write a slat wrong, the reader tells you where and why. Try a broken line on purpose:

(event :ts #inst "not-a-real-timestamp"
       :kind "card.opened")

You’ll get back:

[cart/inbox-log:3:16] #inst tag rejected:
  "not-a-real-timestamp" is not ISO-8601
  did-you-mean: #inst "2026-07-11T09:14:00Z"

The error names the file, the line, the column, and offers a fix. The reader would rather teach you the shape than fail silently. If you meant to write a plain string, drop the #inst tag and it stops complaining.

§L.7 — Signing an incident narrative

Later, when you’re authoring an incident narrative — a small story that teaches Sakura something — you may want to sign it, so that a friend reading it later can verify you actually wrote it.

Signing wraps your slat in a signed envelope:

(signed
  :body      (incident-narrative
               :id "in-1-042"
               :philosopher freud
               :lesson "the tortoise was the missing thought"
               :messages ((role text) ...))
  :signed-by "sakura@lacuna"
  :signature #b64 "…"
  :ts-signed #inst "2026-07-11T18:22:00Z")

The slat/sign verb does the wrapping. The slat/verify verb checks it. You don’t need to know the crypto to use them — you just call the verb and get back #t or a reason. If your key is missing, you get the honest 'service-not-yet-wired escalation — never a fake success.

§L.8 — What to remember

Three things.

  1. A slat is a labeled list. The first word is the label. Everything after is what it carries.
  2. Two file kinds: .slat for one record, .slatl for one record per line.
  3. The reader never runs your slat. A form (if …) reads as three symbols. Data is data.

When you write your next cart that needs to remember something, store something, mail something, or pull something from a mailbox — reach for a slat. The shape is already the shape you know.

Slats are how carts remember. Same shape as Scheme, used for talking instead of doing.
